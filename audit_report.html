AUD-01: Next.js Image Optimization is Disabled Globally [Toggle]
SEVERITY: HIGH
IMPACT: HIGH
CONFIDENCE: 100%
EFFORT: 5 MINS

Description

The configuration file next.config.mjs specifies images: { unoptimized: true }. This disables Next.js's native image compilation pipeline, which automatically resizes, compresses, and generates WebP/AVIF images.

Why it Matters

Disabling image optimization forces the client to download full-resolution source files (like large project screenshots). This increases the page transfer size, degrades Core Web Vitals (Largest Contentful Paint - LCP), and increases mobile network overhead.

Recommended Fix

Remove the unoptimized: true flag and configure standard next/image loaders, or specify domain configuration if using external assets. Since all images are local public assets, next/image will optimize them automatically at build/request time.

Relevant Code

File: next.config.mjs 9: images: { 10: unoptimized: true, 11: },
AUD-02: FAQ Schema sits statically without rendering/embedding [Toggle]
SEVERITY: HIGH
IMPACT: HIGH
CONFIDENCE: 100%
EFFORT: 30 MINS

Description

A structured FAQ database exists at public/faq.json, but it is not embedded via script tags on any route, nor is the FAQ content visually rendered to visitors on any page.

Why it Matters

Google will not crawl or recognize standalone JSON files for structured data (they must be embedded in page HTML). Furthermore, indexing structured schema for content that is invisible to human users violates Google's Webmaster Guidelines, which can lead to manual actions or penalties.

Recommended Fix

Create a dedicated FAQ section on a page (e.g. /about or /contact), parse and display the Q&A list in a clean accordion interface, and dynamically embed the JSON-LD FAQPage schema on that page.

Relevant Code

File: public/faq.json (contains 15 Questions, but never rendered)
AUD-03: Missing Canonical Tags on Homepage and Splash [Toggle]
SEVERITY: MEDIUM
IMPACT: MEDIUM
CONFIDENCE: 100%
EFFORT: 10 MINS

Description

No canonical tags are configured for the root splash page (/) or the main portfolio homepage (/home). While other subpages specify alternates: { canonical: ... }, these critical entryways rely on fallbacks.

Why it Matters

Without canonical declarations, search bots may index multiple duplicate URLs (Vercel deployment subdomains, staging URLs, or www vs non-www paths) leading to diluted link authority and crawler confusion.

Recommended Fix

Inject canonical metadata configuration into the root layout default metadata, or create server-side metadata exports for / and /home to guarantee they reference their canonical URLs.

Relevant Code

File: app/home/page.tsx (use client - metadata not exported) File: app/layout.tsx (metadata generated without canonical)
AUD-04: Invalid Breadcrumb Schema Pathing [Toggle]
SEVERITY: MEDIUM
IMPACT: MEDIUM
CONFIDENCE: 100%
EFFORT: 15 MINS

Description

The static file public/breadcrumbs.json lists all core pages (Home, About, Blog, Gallery, Contact) in a single linear trail with sequential position numbers (1 to 5).

Why it Matters

A breadcrumb trail must represent the hierarchical path of a single page back to the root (e.g. Home > Blog > Post). Listing sibling pages sequentially tells the crawler that Contact is a child of Gallery, which is a child of Blog, representing an invalid site structure.

Recommended Fix

Delete the static, flat breadcrumbs.json file. Instead, ensure each page has its own nested breadcrumb schema injected directly, matching only the actual path of that page (e.g. Blog list has Home > Blog; Blog Post has Home > Blog > Post Title).

Relevant Code

File: public/breadcrumbs.json 4: "itemListElement": [ 5: { "@type": "ListItem", "position": 1, "name": "Home", "item": "..." }, 6: { "@type": "ListItem", "position": 2, "name": "About", "item": "..." }, ...

AUD-07: Missing llms-full.txt Citation Reference [Toggle]
SEVERITY: MEDIUM
IMPACT: MEDIUM
CONFIDENCE: 100%
EFFORT: 15 MINS

Description

The project provides llms.txt at the root of the site, but lacks the accompanying llms-full.txt file, which is an emerging standard for comprehensive AI crawler ingestion.

Why it Matters

Many advanced LLM search and answer engines look for llms-full.txt at the root of a domain. It allows the crawler to fetch the full textual content of the site in a single request, rather than recursively scraping other pages, which can cause incomplete citations if crawling is throttled.

Recommended Fix

Generate a complete llms-full.txt file that appends all relevant blog articles, biography sections, and project logs to the standard llms.txt structure, and place it in the public folder.

Relevant Code

File: public/llms.txt (exists, but llms-full.txt does not)
