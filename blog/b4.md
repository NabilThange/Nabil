# **Why LLMs Don't Actually Remember Anything**

One of the biggest misconceptions in AI is that large language models have memory.

They don't.

An LLM is fundamentally **stateless**.

Each request starts from scratch.

If two identical prompts are sent to the model, it produces responses based solely on that input—not because it remembers previous conversations.

What appears to be memory in applications like ChatGPT is actually an engineering layer that continuously sends relevant context back to the model.

Every conversation history...

Every retrieved document...

Every previous response...

...is simply inserted into the next prompt.

The model itself never stores any of it.

Understanding this distinction is the foundation of building reliable AI agents.

# **The Only Memory Question That Really Matters**

Instead of memorizing seven definitions immediately, start with one architectural question:

**Does the memory exist only during the current conversation, or does it survive beyond it?**

Everything falls into two broad categories:

### **Short-Term Memory**

Lives inside the context window.

Characteristics include:

* Fast access
* Limited size
* Lost after the session ends
* Expensive because tokens are resent every turn

---

### **Long-Term Memory**

Lives outside the model.

Typically stored in:

* SQL databases
* Vector databases
* Knowledge graphs
* Event logs
* Schedulers

Long-term memory survives:

* Crashes
* Restarts
* New conversations
* Multiple users
* Multiple agents

Production AI systems rely heavily on this layer.

Let's examine each one.

---

# **1. Working Memory**

Working memory is everything currently inside the context window.

It includes:

* System prompts
* User messages
* Tool outputs
* Retrieved documents
* Intermediate reasoning

Every LLM call depends entirely on this temporary workspace.

Unfortunately, it comes with several limitations:

* Context limits
* Rising token costs
* Repeated transmission
* Eventual truncation

One overlooked fact is that every new interaction resends much of the previous conversation.

As conversations grow, token costs grow almost quadratically.

This is why relying solely on larger context windows is rarely sustainable.

---

# **2. Semantic Memory**

Semantic memory stores permanent knowledge.

Examples include:

* Customer preferences
* Company policies
* Product information
* Domain knowledge
* User profiles

Unlike conversation history, semantic memory doesn't care when something happened.

It only cares that the information is true.

For production systems, semantic memory is often implemented using:

* PostgreSQL
* pgvector
* Pinecone
* Qdrant
* Weaviate

The biggest challenge isn't retrieval.

It's maintaining a single source of truth as multiple AI agents update shared knowledge.

---

# **3. Episodic Memory**

Episodic memory stores experiences.

Instead of facts, it records events.

Examples include:

* Previous customer interactions
* Fraud investigations
* Completed workflows
* Browser sessions
* AI decisions

Each episode contains:

* Situation
* Decision
* Outcome

When similar situations appear later, agents retrieve comparable experiences before making another decision.

This approach closely resembles case-based reasoning used in cognitive science.

---

# **4. Procedural Memory**

Procedural memory stores **how** something should be done.

Examples include:

* Approval workflows
* Business rules
* Tool usage
* Automation pipelines
* Operating procedures

Unlike semantic memory, procedural memory focuses on execution.

For example:

A claims agent might always follow this order:

1. Validate policy
2. Assess damages
3. Detect fraud
4. Calculate payout
5. Route for approval

Exact rule lookup usually outperforms similarity search here.

Business rules shouldn't depend on vector similarity.

---

# **5. Retrieval Memory Isn't Actually Memory**

Here's the controversial opinion.

Retrieval-Augmented Generation (RAG) isn't really a memory type.

It's a delivery system.

Vector databases don't inherently remember anything.

They simply retrieve stored information based on semantic similarity.

The actual memory lives elsewhere:

* Semantic databases
* Event logs
* Knowledge stores

RAG acts as the transportation layer moving information into the context window.

Thinking of retrieval as memory often leads teams to stop at "we have a vector database."

In reality, they've only built the plumbing.

---

# **6. Parametric Memory**

Parametric memory refers to everything stored inside the model's weights.

This includes:

* Language understanding
* Grammar
* Mathematics
* General world knowledge
* Common reasoning

Unlike external memory, this knowledge:

* Cannot easily be updated
* Has training cutoffs
* May become outdated
* Cannot cite sources

Production systems should rely on parametric memory only for stable knowledge.

Anything dynamic should come from external memory stores.

---

# **7. Prospective Memory**

Prospective memory is remembering to do something later.

Examples include:

* Send a monthly report
* Review a contract next week
* Notify a customer after payment
* Trigger alerts after an event

Unlike semantic memory, nothing is retrieved through similarity search.

Instead, prospective memory relies on:

* Schedulers
* Event queues
* Temporal workflows
* Background workers

Popular technologies include:

* APScheduler
* Celery
* Temporal
* Cron

This capability separates reactive chatbots from proactive AI agents.

# **Building a Production AI Memory Architecture**

Production systems rarely use just one memory type.

Instead, they combine several layers:

| Layer              | Purpose                    |
| ------------------ | -------------------------- |
| Working Memory     | Current reasoning          |
| Semantic Memory    | Persistent facts           |
| Episodic Memory    | Historical experience      |
| Procedural Memory  | Business workflows         |
| Retrieval Layer    | Relevant context injection |
| Parametric Memory  | General intelligence       |
| Prospective Memory | Future actions             |

This layered approach creates AI systems that remain reliable, scalable, and adaptable.

# **Best Practices for Designing Agent Memory**

To build robust AI agents:

* Treat context windows as limited resources.
* Separate facts, events, and workflows into dedicated stores.
* Use retrieval to deliver memory—not replace it.
* Reserve model weights for general knowledge.
* Add schedulers for future tasks.
* Continuously consolidate repeated experiences into reusable knowledge.

One of the most valuable techniques is promoting recurring episodic patterns into semantic rules. This transforms raw experience into lasting knowledge, allowing agents to improve over time rather than simply accumulate logs.

# **Frequently Asked Questions**

### **1. What is agent memory?**

Agent memory refers to the mechanisms that allow AI agents to retain, retrieve, and use information across tasks and conversations.

### **2. How many types of agent memory are there?**

A widely used taxonomy identifies seven types: working, semantic, episodic, procedural, retrieval, parametric, and prospective memory.

### **3. Is RAG considered memory?**

Not exactly. RAG is better understood as a retrieval mechanism that delivers information into an agent's working memory rather than a memory store itself.

### **4. What's the difference between semantic and episodic memory?**

Semantic memory stores general facts and preferences, while episodic memory records specific experiences and past events.

### **5. Why is prospective memory important?**

Prospective memory enables agents to remember future intentions, such as scheduled tasks or event-triggered actions, making them proactive instead of purely reactive.

### **6. Which memory type helps AI agents learn over time?**

The combination of episodic and semantic memory is key. By consolidating repeated experiences into durable facts or rules, agents can improve future decisions.
